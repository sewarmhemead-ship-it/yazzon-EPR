## 1. الفكرة (Vision)

نظام إدارة مخزون بسيط ومستقر لمخبز وكافيه. الهدف العملي:
تتبّع كميات المواد الخام بدقة، وإطلاق **تنبيه شراء تلقائي** بمجرد نزول أي عنصر إلى حدّه الأدنى — ليعرف المدير ماذا يشتري دون متابعة يدوية.

**الأولويات مرتبة:** (1) دقّة الرصيد فوق كل شيء، (2) سرعة التطوير، (3) قابلية التوسّع لاحقاً.

**المستخدمون:** عامل (staff) يسجّل السحب/الإدخال، ومدير (admin) يرى التنبيهات ويدير العناصر.

---

## 2. التقنيات (ثابتة — لا تغيّرها)

- قاعدة البيانات: **PostgreSQL** عبر **Supabase** (قاعدة بيانات + Auth).
- الـ Backend: **Node.js + Express**.
- الوصول للداتا: **node-postgres (`pg`)** عبر connection string. **ممنوع `supabase-js` داخل كود الـ backend.** كل SQL يُكتب يدوياً و parameterized.
- الـ Auth: Supabase Auth للتحقق من الهوية، والصلاحيات في Express middleware.
- **Row Level Security: مُطفأة** — الحماية مسؤولية Express وحده.
- الواجهة: **React + Vite**.
- الاختبارات: **Vitest + supertest** على قاعدة اختبار منفصلة.
- كل الأسرار في `.env` — لا سر واحد مكتوب في الكود.

## 3. مبادئ غير قابلة للكسر (Invariants) — قوانين، لا اقتراحات

هذه القوانين مرقّمة ليُشار إليها في التعليقات المرجعية داخل الكود.

- **[INV-1] لا رصيد سالب أبداً.** كل سحب = UPDATE شرطي ذرّي واحد:
  ```sql
  UPDATE items SET current_stock = current_stock - $1
  WHERE id = $2 AND current_stock >= $1 RETURNING *;
  ```
  صفر صفوف راجعة = رصيد غير كافٍ → ارفض. **ممنوع** نمط "اقرأ ثم انقص" (يعيد الـ Race Condition).
- **[INV-2] كل حركة = معاملة DB واحدة.** تحديث `current_stock` + إدراج صف `transactions` داخل `BEGIN…COMMIT`. ينجحان معاً أو يفشلان معاً.
- **[INV-3] السجل ثابت (Immutable Ledger).** لا حذف/تعديل لأي صف في `transactions`. التصحيح = حركة عكسية جديدة تشير للأصلية.
- **[INV-4] الكميات `numeric`** — لا `float`/`real`.
- **[INV-5] كل التواريخ `timestamptz` (UTC)**، تُعرض بتوقيت `Europe/Vienna`.
- **[INV-6] وحدة أساسية واحدة لكل عنصر** = وحدة الاستهلاك. كل الحركات بها فقط.

## 4. الـ Schema (ثابت)

**users:** `id uuid PK` · `name text` · `email text` · `role text [admin|staff]` · `created_at timestamptz`
**categories:** `id uuid PK` · `name text`
**items:** `id uuid PK` · `name text` · `unit text` · `current_stock numeric` · `min_stock_level numeric` (افتراضي 1) · `category_id uuid FK` · `is_ordered boolean` (افتراضي false) · `last_ordered_at timestamptz null`
**transactions:** `id uuid PK` · `item_id uuid FK` · `user_id uuid FK` · `type text [in|out|waste|adjustment]` · `quantity_change numeric` (موجب=إدخال، سالب=سحب) · `note text` · `created_at timestamptz` (السيرفر) · `reverses_transaction_id uuid FK null`

## 5. الهيكلية الهندسية (Architecture)

بنية **معيارية حسب الميزة (feature-based)** بأربع طبقات صارمة. كل طبقة تعرف الطبقة تحتها فقط:

```
route → controller → service → repository → (DB)
```

- **route:** يربط الـ URL بالـ controller فقط. لا منطق.
- **controller:** يستقبل الطلب ويعيد الرد (HTTP فقط). **ممنوع** أي منطق أعمال هنا.
- **service:** منطق الأعمال + فرض الـ Invariants (هنا يعيش الـ UPDATE الذرّي ولفّ الـ transaction).
- **repository:** استعلامات SQL فقط. لا منطق أعمال.

هذا الفصل هو ما يجعل المشروع **قابلاً للتوسّع**: أي ميزة جديدة تُضاف كطبقاتها الأربع في مجلد مستقل دون لمس البقية.

### قابلية التوسّع (Extensibility) — أين تُضاف الميزات لاحقاً

صمّم الآن بحيث تدخل هذه لاحقاً **دون إعادة بناء**:

| ميزة مستقبلية | أين تُركَّب | لماذا تعمل بسلاسة |
|---|---|---|
| الوصفات/الإنتاج بالدفعات | `modules/recipes/` يستدعي `transactions.service` | أي وصفة = مجموعة حركات `out`. لا تلمس منطق المخزون. |
| صلاحية المنتجات/الدفعات | جدول `batches` + FK من `transactions` | السجل الثابت أصلاً يوثّق كل إدخال بتاريخه. |
| تعدّد الفروع/المخازن | عمود `location_id` على `items` و `transactions` | الاستعلامات تُفلتَر بالموقع دون تغيير المنطق. |
| تقارير واستهلاك | `modules/reports/` يقرأ من `transactions` | السجل الثابت هو مصدر كل تقرير. |

## 6. معايير الكود والتعليقات المرجعية

**التزم بهذه المعايير في كل ملف** حتى يبقى المشروع مقروءاً وقابلاً للصيانة:

1. **ترويسة كل ملف:** تعليق في أعلى كل ملف يذكر: مسؤوليته، طبقته (route/controller/service/repository)، وأرقام الـ Invariants المرتبطة إن وُجدت.
   ```js
   /**
    * transactions.service.js
    * الطبقة: service — منطق أعمال حركات المخزون.
    * يفرض: [INV-1] لا رصيد سالب، [INV-2] معاملة واحدة، [INV-3] سجل ثابت.
    */
   ```
2. **تعليق مرجعي على الكود الحسّاس:** أي سطر يفرض Invariant يحمل تعليقاً واضحاً يربطه بالقانون ويشرح **لماذا** لا يجوز تغييره.
   ```js
   // ⚠️ [INV-1] UPDATE شرطي ذرّي — لا تحوّله إلى SELECT ثم UPDATE، يعيد الـ Race Condition.
   ```
3. **JSDoc على كل دالة عامة:** تشرح **ماذا ولماذا** (لا "ماذا" فقط)، والمدخلات والمخرجات والأخطاء المحتملة.
4. **التسمية:** ملفات `feature.layer.js`، دوال بأفعال واضحة (`consumeStock`, `getLowStockItems`)، لا اختصارات غامضة.
5. **معالجة الأخطاء:** ارمِ أخطاءً من أصناف `shared/errors.js` (مثل `InsufficientStockError`)، ويحوّلها `errorHandler` المركزي لرد HTTP. لا `try/catch` يبتلع الأخطاء بصمت.
6. **لا أرقام سحرية:** الثوابت في مكان واحد بأسماء واضحة.

## 7. سيناريوهات الحياة الواقعية وكيف يتعامل النظام معها

أحداث **تقع فعلاً** في مخبز يعمل، والنظام مُلزَم بالتعامل معها:

| السيناريو | كيف يتعامل النظام |
|---|---|
| شراء الطحين كيس 25 كجم واستهلاكه بالكيلو | وحدة أساسية واحدة (كجم) [INV-6]. استلام الكيس = حركة `in` بكمية 25. لا تُمثّل "الكيس" كوحدة. |
| الجرد الفعلي لا يطابق النظام (هدر، انسكاب، خطأ وزن، سرقة) | حركة `adjustment` تصالح النظام مع الواقع وتبقى موثّقة. النظام ينحرف دائماً — آلية التصحيح إلزامية. |
| نسيان تسجيل توصيلة ثم محاولة السحب | يسجّل التوصيلة بأثر رجعي (`in`) ثم يسحب. لا رصيد سالب تحت أي ظرف. |
| توصيلة ناقصة (23 بدل 25) أو إرجاع فاسد | حركة `in` بالكمية الفعلية أو `adjustment` بالفرق. |
| خبز لم يُبَع / حليب فسد / صينية وقعت | `type = waste` منفصل عن `out` ليرى المدير الخسارة ويقلّلها. |
| الطلب تمّ لكن التوصيلة تأخّرت والتنبيه يتكرر | عند الطلب `is_ordered = true`؛ التنبيهات تفصل "مطلوب" عن "مطلوب وتم طلبه". يعود false عند الاستلام. |
| نزاع: من أخذ آخر كمية؟ | كل حركة مربوطة بـ `user_id` و `created_at`. |
| كميات كسرية (2.5 كجم، نصف علبة) | `numeric` [INV-4]، لا تقريب. |
| العنصر وصل صفر والعامل يريد المتابعة | النظام يوقفه — يجب تسجيل توصيلة أو adjustment. **مقصود**: يكشف النقص بدل إخفائه. |
| بدء استعمال مادة جديدة | إنشاء عنصر سريع من الواجهة. |

## 8. أخطاء شائعة عند البناء بالذكاء الاصطناعي (احذرها)

هذه أخطاء تتكرر ويجب تجنّبها في كل جلسة:

1. **كسر الـ UPDATE الذرّي أثناء "التحسين"** → تحويله لـ SELECT ثم UPDATE يعيد الـ Race Condition. **[INV-1] لا يُمَس.**
2. **فصل تحديث الرصيد عن تسجيل الحركة** → استعلامان منفصلان. يجب أن يكونا في معاملة واحدة. **[INV-2]**.
3. **استعمال `float` للكميات** → أخطاء تقريب. استعمل `numeric`. **[INV-4]**.
4. **خلط `supabase-js` مع node-postgres** → أسلوبان متضاربان. كل الوصول عبر `pg` فقط.
5. **نسيان إطفاء Row Level Security** → يحجب الاستعلامات بصمت.
6. **كتابة الأسرار في الكود** → كلها في `.env`.
7. **انحراف الـ schema** → أي تغيير جداول عبر migration file، لا من لوحة Supabase.
8. **توسيع النطاق تلقائياً** → إضافة جداول/jobs من قسم "خارج النطاق". ذكّر نفسك بالقسم 10.
9. **قبول diffs كبيرة دون قراءة** → خصوصاً في الجلسات الطويلة حين يضعف السياق ويخترع أسماء أعمدة/دوال غير موجودة.

## 9. الاختبارات (إلزامية — بوابة الجودة)

لا تُعتبر أي مرحلة منتهية قبل نجاح اختباراتها. **Vitest + supertest** على قاعدة اختبار منفصلة (`TEST_DATABASE_URL`) تُنظَّف قبل كل اختبار.

الاختبارات الإلزامية:
1. **السحب الصحيح:** ينقص الرصيد ويُدرج صف `out`.
2. **منع الرصيد السالب:** سحب > الرصيد يُرفض والرصيد لا يتغيّر.
3. **الحدّ الصفري:** سحب = الرصيد بالضبط ينجح ويصل صفر.
4. **الإدخال:** يزيد الرصيد ويُدرج صف `in`.
5. **التزامن (الأهم):** سحبان متوازيان (`Promise.all`) على رصيد يكفي لواحد → واحد ينجح وواحد يفشل، والرصيد النهائي صحيح وغير سالب.
6. **الذرّية:** إفشال إدراج الحركة عمداً يجب أن يُرجع الرصيد كما كان (rollback).
7. **الـ Undo:** حركة عكسية تعيد الكمية عبر `reverses_transaction_id` دون حذف/تعديل الأصلية.
8. **التنبيهات:** الاستعلام يُرجع بالضبط العناصر `current_stock <= min_stock_level`.
9. **دقة الأرقام:** كميات كسرية (0.1، 0.2، 2.5) بلا أخطاء تقريب.
10. **الصلاحيات:** `staff` لا يصل لمسارات `admin`.

**البوابة:** `npm test` يجب أن يمر بالكامل، واعرض مخرجات النجاح. اختبار فاشل = المرحلة غير منتهية.

## 10. خارج النطاق (لا تبنِها إلا بطلب صريح)

توثيق يومي / تسكير يوم / لقطات يومية · جدول PurchaseRequests و jobs دورية (التنبيه استعلام لحظي) · الوصفات/الإنتاج بالدفعات · صلاحية المنتجات · وضع Offline-first.
بناء أيٍّ منها دون طلب = خطأ يُتراجَع عنه.

**منطق التنبيه** (استعلام لحظي، لا jobs):
```sql
SELECT * FROM items WHERE current_stock <= min_stock_level
ORDER BY is_ordered ASC, (min_stock_level - current_stock) DESC;
```

## 11. خطة التنفيذ (مرحلة واحدة في كل مرة، قف للمراجعة بعد كل واحدة)

1. **Base Setup:** migrations للجداول + `config/db.js` + هيكل Express الأساسي. أوقف واعرض الـ migrations قبل أي كود تطبيقي.
2. **Auth & RBAC:** ربط Supabase Auth + middleware الأدوار.
3. **Core Logic:** `addStock` / `consumeStock` (UPDATE ذرّي داخل transaction) / `undoTransaction` + اختباراتها (القسم 9).
4. **Alerts API:** مسار التنبيهات + منطق `is_ordered`.
5. **UI (React):** قائمة العناصر بأزرار (+/−)، شاشة التنبيهات، إنشاء عنصر، Undo لآخر حركة.

**تعريف "تمّ" لكل مرحلة:** الكود يلتزم بالقسم 3 · اختباراتها تمر · لا أسرار في الكود · عرضتَ ملخصاً + مخرجات الاختبارات + المرحلة التالية.
